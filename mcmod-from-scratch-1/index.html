一切的开始

Forge 与 FML

Forge从来没有宣称过其类型，但是根据其功能来看，其更像是一套框架。

Forge包含两个部分：Minecraft Forge和Forge Mod Loader (FML)。

FML提供了整合多个模组的机会。其介入了MC的底层，能让你在Minecraft中一口气装一大堆模组，在你的电脑撑不住前想装多少装多少。

Minecraft Forge本身提供了两套东西：

  1. 对Minecraft底层的修改及因此暴露出的公开方法。当有多个模组因为同时修改Minecraft的某一个部分而互相冲突时，Forge可以介入Minecraft的底层来提供一套令这些模组不再冲突的解决方案。
  2. 一些相对来说独立于MC之外的系统。这些系统看起来和MC之间没有多少耦合，但能极大改善模组之间的兼容性，代表系统有流体、事件总线和矿物词典。这些小系统比其他Forge的组成部分更接近真正的API。

MCP

众所周知，MC是一款商业性软件，用脚想都可以知道的一件事情就是我们不可能获取MC的源代码。
不过由于Java的编译器编译的不是那么彻底，我们还是可以通过逆向编译获取一些源代码的结构的。
当然这些逆向编译之后的代码可读性接近0，其类名、变量名、方法名都是经过混淆之后的产物，诸如Item这一类好读又好记的名字全部被替换成了诸如cks等毫无意义的东西。
当然，有一些地方会有一些提示，比如字符串常量、异常信息等等。
这就给了一些人打开MC源码大门的窗口。
在经过一群社区贡献者的努力之后，一套完整的反混淆映射表出厂了。这个映射表被称作”MCP mapping”。

当然，这套表也在随着MC的更新而不断更新。
每一天美国东部时间的凌晨三点，MCP都会发布一个快照版本，以当日的日期命名。
而当MCP决定放弃当前版本，转而开发下一个版本的时候，会发布一个稳定版。

某一个时间点之后，微软官方开始发行名为“官方版本”的Mapping。每一个版本发行一次，以版本号命名。

对于1.16.5版本来说，Forge使用的是官方版本，official_1.16.5。
不过不用担心，上面的快照和稳定两版还在开发。

MC的源码经过MCP反混淆之后，可以转化为我们能看懂的代码，而我们写的代码也会经过MCP再混淆之后并入MC。

主类和Mod信息

主类

首先迎面而来的是一个叫做主类的东西。

Forge已经帮你建立好示例了，就在src\main\java\com\example\examplemod\examplemod.java里面。

首先你需要选择一个好的包名，比如说我的就叫KaiserWilheim.Kaiserdom。
示例的代码的包名叫做com.example.examplemod，也就是把路径中java后面的东西直接放上去，然后把文件结构分隔符改为.就可以了。

拿捏不定怎么命名的，可以去GitHub上开源Mod的仓库看看文件夹结构，或许有帮助。

配置项 是否必选 作用
modLoader 是 选择加载模组的工具，不用改，使用默认即可。
loaderVersion 是 加载工具的版本，也是默认即可。
license 是 许可证，建议不好选的看这里。
issueTrackerURL 否 你的模组的Bug提交地址。
modId 是 你的模组的ID。只能使用小写字母和下划线。
version 是 版本号。建议根据这里取一个标准一点的版本号。
displayName 是 展示在Mods界面内的模组名。
updateJSONURL 否 你的模组的更新连接。
displayURL 否 你的模组的官网。
logoFile 否 你的模组的logo。
credits 否 你的模组的致谢名单。
authors 否 你的模组的制作者名单。
description 是 你的模组在Mod界面的介绍。
